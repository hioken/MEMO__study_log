# オブジェクト
## CDN(Content Delivery Network)
### 概要
* 物理的な遅延を緩和
### 機能
* オリジンサーバー: オリジンサーバーはどこか一つのDC
* エッジネットワークの分散: CDNプロバイダが、世界中の主要都市に無数のエッジノード(中継サーバー)を配置
* コンテンツのキャッシュと代理応答: 静的アセットをオリジンサーバーから各エッジノードにキャッシュしておく
* Edgeノードの軽量ランタイム: V8エンジンをベースとした、軽量な実行環境が稼働可能

# インフラ配置
## rails(AWS etc)
### 配置
* 特定のDCにあるサーバー内にappを配置
### プロセス/ステート
* AppServerは年中無休でプロセスがメモリ上に常駐する
* 外部ツールに頼らずともステートフルなセッション管理等が行える
* 常時TCPをDBサーバーへ繋げられる
### リクエスト受信
1. Nginx等のリバースプロキシ
2. Puma等のAppServer
3. メインプロセス

## Next.js(Vercel etc)
### 配置
* インフラの配置場所が特定のDCに限定されていない
* メインサーバーに加えて、CDNにデプロイされる
### プロセス/ステート
* リクエストを受信すると起動する
* 同時に複数リクエストが届いても、必ず1リクエスト1実行環境を立ち上げる
* プロセスが頻繁に終了するため、外部ツールに頼らないとステートフルなセッション管理等は行えない
* 外部ツールのコネクションも常時繋げておけない問題が発生し、外部コネクションプーラー等の解決法が必須
### リクエスト受信
1. Edgeノードでのインターセプト: Middlewareの起動
2. ルーティング前のリクエスト検証: Middlewareの処理
3. レスポンス: Middlewareの処理範囲内ならそのままレスポンスを返す、そうでないならメインサーバーへ通す
### ステートレス
#### 外部コネクションプーラー
###### 概要
* appとdbの間に、dbとの接続を維持する中間プロキシ層をインフラとして追加する
###### 機能
* DBとの間にTCPを常時少数繋げておく
* 上記の接続内で処理を行えるようなキューイング
* および、複数のTCP経路を用いた効率的な並列処理

# デプロイ
## Rails+AWS
### 配置
* 全てAppServerに集約されている
### デプロイ
* 対象サーバーでプロセスを実行
## Next.js+Vercel
### 配置
#### エッジネットワーク(CDN)
* 静的アセット、事前ビルド済みファイル
* エッジロジック(Middleware.js)
#### メインリージョン
* Node.js実行環境
* Appコード
### デプロイ
1. 中央ビルトサーバー: Vercelが用意したビルド専用の一時サーバーでビルド工程
2. 分割(`Client`, `Edge+StateLess`, `Server`)
3. 分散デプロイメント: それぞれの箇所にデプロイ
4. トラフィックの切替: 全てのノードと環境への配置
